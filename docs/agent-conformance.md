# Agent conformance — the 2026-07-09 field test and what it changed

## Setup

Internal hackathon, 2026-07-09. Two teams, each using **Claude Code** with (a) this repo, (b) a
hackathon example set (neutralised `plant.xh` lab; 9 machines, profile+source+sync per machine)
and (c) briefing slides introducing a FabricationOrder challenge. Same live sources for both:
OPC-UA (REST shim), a PPS clone (PostgREST) and an ERP sim. Both teams wrote into their own
Neo4j + TimescaleDB via a structured write API. One day.

## Result

Two complete factory graphs (12,758 nodes, 16,692 edges combined) — **mutually incompatible**:
three names for the same edge semantics (`ON_MACHINE` / `AT_WORKCENTER` / `RUNS_ON`), machine
identity modelled three ways, zero shared merge keys.

Measured with `ci/conformance.mjs` (nodes: declared label + its key property present; edges:
declared triple):

| measured against | team1 nodes | team1 edges | team2 nodes | team2 edges |
|---|---|---|---|---|
| hackathon-set contract (element_id world) | 0.0 % | 0.0 % | 0.0 % | 0.0 % |
| **this repo's contract** (machine_id world) | **62.3 %** | **43.1 %** | 0.0 % | 0.0 % |

## The honest diagnosis

1. **We shipped diverging sources of truth.** The repo says `ProductionOrder`, machines key on
   `machine_id`, `RESPONDS_TO`/`ON_MACHINE` are canonical. The hackathon set says `CNC_Machine`
   keys on `element_id`. The slides said `FabricationOrder` + `Serial`. Each agent locked onto a
   different one (team1 → repo, team2 → slides). No agent could be conformant, because
   "conformant" was not defined in one place.
2. **Nothing enforced anything.** The write API answered `200 OK` to every invented label and
   edge. Agents correct themselves on error messages, not on documentation.
3. **Descriptive schemas do not bind agents.** The normative facts (labels, keys, edges) were
   spread over dozens of prose-heavy files; several machine profiles declared no relationships at
   all. LLM priors filled the gaps.

## Consequences (conformance levels)

- **L1 — files valid** (existing linters + shape validation): necessary, proven insufficient.
- **L2 — writes conformant**: `contract.json` (generated by `ci/gen-contract.mjs`, drift-checked
  by `ci/lint-contract.mjs`) + `CLAUDE.md` as the agent instruction + sink-side validation
  (unknown label/edge → 422 naming the correct form) + `ci/conformance.mjs` as the benchmark.
- **L3 — identity continuous**: one equipment key across OT and IT. Open consolidation:
  `machine_id` (repo profiles) vs `element_id` (UNS/lab convention) — tracked in
  `contract.json → identity.openConflict`. Until resolved, L3 cannot be met.

The 2026-07-09 graphs are preserved as the benchmark baseline. Every enforcement measure must
move the conformance score measurably; rerunning the same task with contract + enforcement is
the acceptance test ("round 2").

## Open follow-ups

- Resolve the machine-key conflict (decide: `element_id` everywhere, or document `machine_id`).
- Resolve the 7 `unresolvedTargets` in `contract.json` (equipment hierarchy levels such as
  `EquipmentUnit` are edge targets but have no node profile).
- Add a serialized-part profile (the trace level under the order — the strongest edge in team2's
  graph, 827× `Serial -PRODUCED_BY-> FabricationOrder`, currently has no home in this repo).
- Sink-side 422 validation in the lab write API.
